String 类的常见方法

StringBuffer类

代表可变字符序列,

  • StringBuffer的==父类有属性char[] value;并不是final==
  • (不是final的)value数字存放字符 —> 存在堆中
  • StringBuffer是final的
  • StringBuffer字符串内容在变化的时候不用不用每次都更换地址(不用每次创建对象),char[] value要比存的字符串长一点,要追加的话直接跟上就行了,太长的时候才会创建新的对象,重新指向新的char[] <看下面图>

String VS StringBuffer

StringBuilder

继承关系图:

  1. 一个可变的字符序列.此类提供一个与StringBuffer兼容API,但不保证同步(StringBuider不是线程安全的).此类被实际用作StringBuffer的一个建议替换,==用在字符串缓冲区被单个线程使用的时候==.如果可能,建议优先采用该类,因为在大多数实现中,它比StringBuffer要快
  2. 在StringBuilder上的主要操作是append与insert方法,可重载这些方法,已接受任意类型的数据

Stringbuilder与StringBuffer均是代表可变的字符序列 ,方法是一样的

  1. StringBuilder 是final类,不能被继承

  2. Stringbuuilder 对象字符序列仍然是存放在其父类中的.

    AbstractStringBuilder的 char[ ] value;

    因此,字符序列是存在堆中的

  3. StringBuilder 的方法,没有做互斥处理,即没有synchronized 关键字.因此推荐在==单线程的情况下,使用Stringbuilder==

StringBuffer与StringBuilder比较

  1. String : 不可变字符序列,效率低,但是复用率高

  2. StringBuffer : 可变字符序列,效率较高(增删).线程安全

  3. StringBuilder : 可变字符序列,效率最高,线程不安全

  4. String 使用注意说明:

    String s=”a”//创建了一个字符集

    s += “b”//实际上原来的”a”字符串对象已经丢弃,现在又船身了一个字符

    串s + “b” (也就是”ab”).如果多次执行这些改变串内容的操作,会导致大量的副本字符串对象残留在内存中,降低效率.如果这样的操作放到循环中,会极大影响程序的性能 => 结论: 如果我们对String 做大量修改,不要使用String,应该使用StringBuffer,如果是单线程的话,更推荐使用StringBuilder

==效率== : StringBuilder > StringBuffer > String
